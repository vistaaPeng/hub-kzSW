# 天气查询「链式 Function Call」改造总结

> 作业目标:把「查天气」函数拆成**查经纬度**和**查天气**两个函数,再把函数调用部分改造成**链式调用** —— 前一个工具的结果成为下一次调用的输入,多轮闭环自动串成调用链。
>
> 改造后可查询**北京、上海、天津**的经纬度和天气。

## 二、核心改造点

### 改造点 1:一个函数 → 两个单一职责函数

**改造前**(`function_call_mcp_cli/src/weather_backend.py`):
`get_weather(city)` 一个函数内部做了两件事 —— 先 Geocoding(城市名 → 经纬度),再查天气(经纬度 → 天气)。两步耦合在一起,外部看不到中间的经纬度。

**改造后**(`homework/weather_backend.py`)拆成两个:

| 函数 | 输入 | 输出 | 职责 |
|------|------|------|------|
| `get_coordinates(city)` | 城市中文名 | **JSON 字符串**,含 `latitude` / `longitude` / `name` 等 | 只做 Geocoding |
| `get_weather(latitude, longitude, name="")` | 经纬度 | 天气文字报告 | 只做天气查询 + 格式化 |

**关键设计**:`get_coordinates` 返回 **JSON 字符串**,因为它的结果要回填给模型,模型再从中取出 `latitude` / `longitude` 作为 `get_weather` 的入参 —— **这正是"链条的接缝"**。

> 拆分时保留了原有的「宁德」地名歧义消除逻辑(优先地级市而非同名小村庄),以及"错误返回可读字符串而非抛异常"的风格。

### 改造点 2:单轮闭环 → 多轮闭环

**改造前**:`run()` 只跑**单轮**闭环 —— 模型一次性输出所有 `tool_call`,宿主执行后回填,模型立刻产出最终回答。这种结构下,**一个工具的结果无法成为另一个工具的输入**。

**改造后**:把它改成 **`while` 多轮循环**:

```
while 轮次 < max_rounds:
    请求模型(带 tools)
    if 模型不再要求调用工具:
        break            # 本轮即最终回答
    逐个执行 tool_call,把结果以 role=tool 回填
    → 进入下一轮,模型读到结果后决定下一步
```

加了 `max_rounds=6` 作为防死循环的兜底。

## 三、链式调用是怎么"自动"串起来的

以「天津现在的经纬度和天气如何?」为例,多轮闭环自动形成 3 轮调用链:

```
第 1 轮  模型 → get_coordinates("天津")
              ↩ {"latitude": 39.14, "longitude": 117.18, "name": "天津", ...}

第 2 轮  模型读到经纬度 → get_weather(39.14, 117.18, "天津")   ← 用了第 1 轮的结果!
              ↩ 【天津】天气报告 当前阴天 30.7°C ...

第 3 轮  模型读到天气 → 产出最终自然语言回答(不再调用工具)→ 跳出循环
```

**要点**:开发者没有硬编码"先调 A 再调 B"的流程。是**模型**根据两个工具的 schema 描述(`get_weather` 的经纬度"必须来自 get_coordinates")+ System Prompt 的引导,**自主**地把两次调用串成了链。宿主只负责一件事:每轮执行工具、回填结果、再问一次模型。

## 五、验证结果

### 5.1 单城市查询（链式:1 城市 = 2 次串联调用）

| 城市 | 经纬度(get_coordinates) | 天气(get_weather) | 链路 |
|------|------------------------|-------------------|------|
| 北京 | 39.91°N, 116.40°E | 正常返回 | ✅ 2 次调用自动串联 |
| 上海 | 31.22°N, 121.46°E | 正常返回 | ✅ 2 次调用自动串联 |
| 天津 | 39.14°N, 117.18°E | 正常返回 | ✅ 2 次调用自动串联 |

三个城市均验证通过,`get_coordinates` 的输出成功作为 `get_weather` 的输入被模型串联使用。

### 5.2 多城市一次查询(并行 + 链式组合)

问题:`汇总北京、上海、天津三地的经纬度和天气`

一条问题里查 3 个城市,多轮闭环自动形成如下轨迹——**同一轮内并行、轮与轮之间链式**:

```
第 1 轮  模型并行调用 3 个 get_coordinates(北京 / 上海 / 天津)
           ↩ 北京 39.91,116.40 | 上海 31.22,121.46 | 天津 39.14,117.18
第 2 轮  模型并行调用 3 个 get_weather,分别用上一轮返回的经纬度   ← 跨轮链式
           ↩ 北京阴天 30.2°C | 上海阴天 31.1°C | 天津阴天 29.9°C
第 3 轮  模型汇总 3 地结果,产出最终回答(不再调用工具)→ 结束
```

要点:

- **同轮并行**:一轮里模型可一次发起多个 `tool_call`(3 个城市的经纬度一起查),宿主逐个执行后一并回填。
- **跨轮链式**:第 2 轮的 `get_weather` 参数,全部来自第 1 轮 `get_coordinates` 的返回,由模型自主传递。
- 两者叠加:多轮闭环既支持"广度"(并行多工具),又支持"深度"(链式依赖),3 个城市 × 2 步 = 6 次工具调用,3 轮跑完。

### 5.3 中文/emoji 显示

早期在中文 Windows 控制台(GBK 码页)输出含 emoji 的回答时,会出现地名尾字重复、emoji 损坏(如 `📍上海`→`�📍上海海`)。已在两个脚本开头加入 `sys.stdout.reconfigure(encoding="utf-8")` 强制 UTF-8 输出修复;VS Code 终端按 UTF-8 解码即正常显示,无需再设 `chcp` 或 `PYTHONIOENCODING`。

## 六、单轮 vs 多轮 对比小结

| 维度 | 单轮闭环(原) | 多轮闭环(改造后) |
|------|-------------|-----------------|
| 循环结构 | 固定两次请求 | `while` 循环,轮次由模型决定 |
| 工具间数据传递 | ❌ 不能(结果只喂给"最终回答") | ✅ 前一工具结果可作后一工具入参 |
| 适用场景 | 并行、互相独立的工具调用 | 有依赖关系的**链式**工具调用 |
| 防护 | 无需 | `max_rounds` 防死循环 |
