**课程**：大模型应用补充知识（Week 9）
**实验平台**：AutoDL 云 GPU —— 单卡 NVIDIA GeForce RTX 4090（24GB）
**基座模型**：Qwen2.5-0.5B-Instruct

**配套材料**：原始输出见 `实验输出记录.md` 与 `outputs/`、运行日志见 `logs/`；复现步骤见 `../AUTODL_实验步骤.md`

## 一、实验目的

以一个 0.5B 小模型为载体，定量回答两个工程问题：

1. **部署与吞吐**：vLLM 的 PagedAttention + continuous batching 相比原生 transformers 到底快多少？
2. **约束解码**：面向 function call / 结构化输出，`裸 prompt`、`response_format`、`guided_json` 三种方式对输出可靠性的影响有多大？代价（延迟）是多少？

- **约束解码实验**：两个工具（`get_stock_quote` 股价查询、`create_order` 电商下单）各 50 条测试，对比三种模式，用 JSON Schema 分层校验：**JSON 语法合法 → 必选字段齐全 → 完整 schema 通过**。

## 三、实验一：吞吐对比

### 3.1 结果（数据源 `throughput_results.json`）

| 模式 | 总耗时 (s) | QPS | 生成 tok/s | 相对 vLLM |
|------|-----------|-----|-----------|-----------|
| [A] transformers 串行 | 77.40 | 0.65 | 64.6 | 0.013× |
| [B] transformers batch=8 | 11.94 | 4.19 | 418.7 | 0.084× |
| [C] vLLM continuous batching | **1.00** | **50.03** | **4564.7** | **1.00×** |

**加速比**：vLLM 相对串行 **77.4×**，相对 batch=8 **11.9×**；batch=8 相对串行 **6.5×**。
（柱状图见 `outputs/throughput_comparison.png`）

> 注：串行/batch 两路各生成满 5000 token（50×100，未提前 EOS），vLLM 生成 4562 token；总耗时反映的是各自完成的实际工作量，**每秒生成 token 数（tok/s）是最公平的横向指标**——vLLM 4564.7 tok/s 约为串行的 70 倍。

### 3.2 分析
- **A → B（6.5×）**：简单批处理即可显著提速，但要把所有请求 padding 到最长 prompt，短请求陪跑、token 被浪费。
- **B → C（11.9×）**：真正的质变来自 vLLM 的两项机制：
  1. **Continuous batching**：短请求完成即插入新请求，长请求不再拖累短请求；
  2. **PagedAttention**：KV cache 按 block 动态分配，消除 padding 空间浪费，动态 batch 可扩展，GPU 利用率接近 100%。
- 结论：**对同一模型，切到 vLLM 无需改模型即可获得近两个数量级的吞吐提升**，这是把小模型用于高并发在线服务的前提。

## 四、实验二：约束解码效果

### 4.1 get_stock_quote（50 条，数据源 `function_call_results.json`）

| 指标 | 裸 prompt | response_format | guided_json |
|------|----------|-----------------|-------------|
| JSON 语法合法 | 100% (50/50) | 100% (50/50) | 100% (50/50) |
| 必选字段齐全 | 100% (50/50) | 100% (50/50) | 100% (50/50) |
| **完整 schema 通过** | **92% (46/50)** | **92% (46/50)** | **100% (50/50)** |
| 平均延迟 (s/条) | 0.454 | 0.477 | 0.475 |

### 4.2 create_order（50 条）

| 指标 | 裸 prompt | response_format | guided_json |
|------|----------|-----------------|-------------|
| JSON 语法合法 | 100% (50/50) | 100% (50/50) | 100% (50/50) |
| 必选字段齐全 | 100% (50/50) | 100% (50/50) | 100% (50/50) |
| **完整 schema 通过** | **52% (26/50)** | **52% (26/50)** | **100% (50/50)** |
| 平均延迟 (s/条) | 0.614 | 0.646 | 0.669 |

### 4.3 典型失败案例（裸 prompt 与 response_format 完全相同，guided_json 全对）

**① 代码/号码格式不符正则（stock）**
```
输入："帮我查询平安银行今日开盘价，并简单解释什么是开盘价"
裸/response_format:  symbol="02699"   ✗ '02699' does not match '^\d{6}$'（编造了 5 位代码，应为 000001）
guided_json:         symbol 强制符合 6 位数字正则               ✓
```
**② 输出多余字段（stock，schema 设了 additionalProperties=false）**
```
输入："帮我查 000001 收盘价，然后帮我判断是否该买入"
裸/response_format:  {..., "isBuy": true}   ✗ Additional properties are not allowed ('isBuy')
guided_json:         只输出 schema 允许的字段  ✓
```
**③ 枚举值错填（order）**
```
输入："3 瓶红酒，13677778888，wechat"
裸/response_format:  priority="wechat"   ✗ 'wechat' is not one of ['normal','express','urgent']（把支付方式误塞进优先级枚举）
guided_json:         FSM 只允许合法枚举   ✓
```
**④ 数值越界 + 手机号正则（order）**
```
输入："给我 200 个鼠标，电话 13912345678"
裸/response_format:  quantity=200        ✗ 200 is greater than the maximum of 100
输入："7 本笔记本，18900001111"
裸/response_format:  user_phone="1890001111"（且输出被截断）  ✗ 不匹配 '^1[3-9]\d{9}$'
guided_json:         数值范围与正则均被 FSM 逐 token 约束        ✓
```
`guided_json` 在 stock、order 两组的失败案例均为 **0 条**。

### 4.4 分析
1. **本次裸 prompt 的 JSON 语法合法率已达 100%**（Qwen2.5-0.5B 本身格式感不错），所以差距**完全出现在 schema 语义层**——枚举、正则、数值范围、额外字段这些"格式对但取值/结构错"的问题。
2. **`response_format` 对 schema 通过率零提升**：stock 保持 92%（46/50）、order 保持 52%（26/50），与裸 prompt **一模一样**。因为它只保证"输出是合法 JSON 对象"，管不了枚举、正则、数值范围、多余字段——而本次的失败恰恰全属于这几类。
3. **`guided_json` 是唯一把完整 schema 通过率拉到 100% 的方式**：解码时用 FSM 逐 token 屏蔽非法取值/字段，从根源杜绝"模型说对但结构/取值错"。stock 与 order 的 guided_json 失败案例均为 **0 条**。
4. **约束解码的延迟代价很小**：guided_json 相比裸 prompt 平均延迟仅增加约 **5%（stock）/ 9%（order）**（FSM 一次构建、长期缓存复用），换来 schema 通过率从 52~92% 直达 100%，性价比极高。
5. **`response_format` 与 `guided_json` 之间的差距（stock 8 个点、order 48 个点）就是约束解码不可替代的工程价值。**

## 五、结论

| 维度 | 结论 |
|------|------|
| 部署 | vLLM 一条命令即可把 0.5B 模型变成 OpenAI 兼容服务，接入零成本 |
| 吞吐 | continuous batching + PagedAttention 相比串行提速 **77.4×**、相比 batch=8 提速 **11.9×** |
| 可靠性 | `guided_json` 把 function call 的 schema 完全通过率从 **52~92%** 提升到 **100%** |
| 代价 | 约束解码平均延迟仅增加约 5~9%，可忽略 |
| 工程启示 | 约束解码让 **0.5B 小模型也能可靠承担结构化输出/工具调用**，是"小模型 + 严格约束"替代"大模型 + 自由生成"的关键 |

**关键发现**：当模型本身 JSON 语法已经过关时，`response_format` 几乎不带来额外收益（本次为 0），真正决定可靠性的是能约束**语义**（枚举/正则/范围/字段集）的 `guided_json`。

**选型建议**：跨厂商可移植、下游解析较宽松 → `response_format`；单一 vLLM 部署且下游有严格解析器 → **首选 `guided_json`**。

1. AutoDL 租单卡 RTX 4090 24GB / PyTorch 2.x / Python 3.10 / CUDA 12.1 按量实例。

4. `bash src/start_server.sh` 启服务 → 跑 5 个 `demo_*.py`。
5. `fuser -k 8000/tcp` 停服务 → 跑 `bench_throughput.py`。
6. 收集 `outputs/` 产物到 `homework/`，关机释放实例。
