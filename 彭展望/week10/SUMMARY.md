# 第十周作业 — 最简单的检索增强问答（RAG）demo

> 作业要求：基于手头文件写一个问答系统，不需要网页，能用代码做出回答就行，方法任选。
> 完整代码见同目录 [`qa_demo.py`](./qa_demo.py)（单文件，无需向量库/向量模型）。

---

## 一、做了什么

在贵州茅台2023年年度报告上做了一个最小可用的 RAG 问答：

1. **切片**：复用课程素材里已经解析好的 PDF JSON（`../week10 检索增强生成RAG/rag_annual_report/data/parsed/`），
   按原始 block 切片——表格独立成片，短标题会并入紧随其后的表格/正文，避免"标题"和"内容"拆成两片后检索只命中标题。
2. **检索**：BM25（`rank_bm25`）+ `jieba` 分词，只保留长度 ≥2 的词（过滤掉"的/是/了"之类单字虚词）。
   起初直接用整段字符 2-3gram 的 TF-IDF，"贵州茅台/2023/年"这类在财报里反复出现的高频词会把只出现一两次
   但真正相关的关键词（比如"净利润"）盖过去，换成 jieba 分词 + 过滤单字虚词后检索准确率明显变好。
3. **生成**：把 Top-5 片段和问题一起交给 DeepSeek（`deepseek-chat`，OpenAI 兼容接口），要求"只根据参考资料回答，
   不要编造"。没有配置 API Key 时会自动降级为直接展示检索到的最相关原文片段，所以整条链路在没有 key 的
   情况下也能跑通，不依赖网络。

## 二、环境

| 项 | 值 |
|----|----|
| 文档 | 贵州茅台2023年年度报告（PDF 解析 JSON，948 个原始 block → 616 个检索片段） |
| 检索 | BM25Okapi（rank_bm25）+ jieba 分词，纯 CPU，无需下载任何模型 |
| 生成 | DeepSeek `deepseek-chat`（OpenAI 兼容接口），key 放在本地 `.env`（已加入 .gitignore，不提交） |

## 四、如何复现

```bash
cd qa_demo 目录
pip install -r requirements.txt

python qa_demo.py                                  # 跑内置的 4 个示例问题
python qa_demo.py -q "贵州茅台2023年的净利润是多少？"  # 问自定义问题
```

`DEEPSEEK_API_KEY` 已经写进 `~/.zshrc`（各项目通用，不再放在项目里的 `.env`），新开终端就能直接用；
换一台机器/换环境的话可以 `cp .env.example .env` 填自己的 key（或 `DASHSCOPE_API_KEY` / `OPENAI_API_KEY`），
两种方式脚本都认。不配置任何 key 也能跑，只是回答会退化成"展示检索到的原文片段"而不是 LLM 总结的一句话。

## 五、一点体会

这周踩的坑主要是检索这一步，而不是"要不要用向量数据库"：

- 财报 PDF 解析后，表格里的中文标签经常被单元格换行切断（比如"净利 润"中间多一个空格/换行），
  检索前把空白字符全部去掉再分词，能找回不少本来匹配不上的关键词。
- 字符级 n-gram 的 TF-IDF 对中文财报这种"实体名 + 年份反复出现"的文本不太友好，换成分词 + BM25，
  并过滤掉单字虚词后效果提升明显——这也是为什么真实项目里通常还要再叠一层向量检索 + rerank
  （课程素材里的 `rag_annual_report` 就是这么做的），单纯 BM25 在语义匹配、同义词（比如"营收" vs "营业收入"）
  上还是会有局限，只是对这份作业的几个问题够用了。
