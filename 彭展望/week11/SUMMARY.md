# 第十一周作业 — 把天气查询的工具调用改造为循环调用

> 作业要求：把天气查询的工具调用改造为循环调用。
> 完整代码见同目录 [`weather_loop_demo.py`](./weather_loop_demo.py)。

---

## 一、课堂代码的问题

课堂素材 `function_call_mcp_cli/mode_function_call/run_function_call.py` 里的 `run()` 是"单轮闭环"：

1. 请求一次，模型可能返回 `tool_calls`
2. 执行工具、把结果回填
3. **固定再请求一次**拿最终回答，第二次请求即使模型还想调工具也不会再执行

这种写法在"参数一次能摆齐"的场景没问题（比如同时问三个城市天气，模型一轮里并行发
三个 `tool_call`），但遇到"要先看到上一次工具结果才能决定下一步查什么"的场景就不够
用——模型明明还想再调一次工具，却被结构强制去憋一个最终回答。

## 二、改造方式

把"固定跑两次请求"改成 `while True` 循环：只要模型返回 `tool_calls` 就执行 + 回填 +
继续下一轮请求，直到模型主动不再要求调用工具为止，另加 `MAX_ROUNDS=5` 兜底防止模型
异常时无限循环。核心差异：

```python
# 单轮闭环（课堂代码）：只允许「请求→执行→请求」跑一次
resp = create(...); if msg.tool_calls: 执行+回填; resp = create(...)  # 第二次不再检查 tool_calls

# 循环调用（本作业）
while True:
    resp = create(...)
    if not msg.tool_calls:
        return msg.content          # 模型主动停下才退出
    执行 + 回填
    # 不 return，回到循环顶部，让模型自己决定还要不要继续调
```

工具本身沿用课堂的 `src/weather_backend.py::get_weather`，没有重复实现。

## 三、验证效果（原始输出见 `run_output.log`）

用一个**必须分两步决策**的问题验证循环是否真的生效：

> 先查一下哈尔滨现在的天气，如果气温低于10度，就再帮我查一下三亚的天气，我要决定去哪
> 避寒；如果哈尔滨不冷就不用查三亚了。

实际跑出来是 **3 轮**（比预想的还多一轮，正好验证了循环的另一个好处）：

| 轮次 | 发生了什么 |
|------|-----------|
| 第1轮 | 模型调用 `get_weather(哈尔滨)`，但接口刚好 SSL 握手超时，返回了错误文本 |
| 第2轮 | 模型看到工具返回的是错误信息，**自己决定重试**同一个工具调用 |
| 第3轮 | 拿到"哈尔滨 25.7°C"后，模型判断不冷，直接给最终回答，**没有再调三亚** |

即：网络失败自动重试、以及"哈尔滨不冷就不查三亚"的条件分支，两种都是单轮闭环版本
做不到的——单轮闭环遇到第一次报错或者模型想追加调用时，会被结构直接打断去生成"最终
回答"，只能编出一句不准确的话。

对照组问题（`北京、上海、宁德今天天气怎么样？`）只需要 2 轮：第1轮模型一次并行调用
3 个城市，第2轮直接给出最终回答——说明改造后的循环对"一次能问完"的场景没有多余开销，
只在真正需要多步决策时才会多跑几轮。

## 五、一点体会

一开始以为"循环调用"只是把 if 改成 while 这么简单，跑起来才发现它顺带解决了一个没
预料到的问题：工具调用失败时的重试。因为错误信息也是以 `role=tool` 正常回填进对话的，
模型看到"天气数据获取失败"这句话，会把它当成一个需要处理的结果，自己决定要不要重试
——而不是被单轮闭环的结构逼着直接把这条报错原样包装成最终回答讲给用户听。这也是为什么
真实的 Agent 框架（比如 ReAct）都是循环结构而不是固定轮次，单轮闭环本质上只是"循环
的特例（循环一次）"。
