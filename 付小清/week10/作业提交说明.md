# 第十周作业提交说明

## 一、作业题目

基于手头文档构建一个问答系统，能用代码做出回答即可，方法任选。

## 二、我的方案

构建了一个 **医学科普 RAG（检索增强生成）问答系统**：

- **知识库**：自建 13 篇 Markdown 医学科普文档（7 种常见疾病 + 5 种症状指南）
- **技术路线**：文档分块 → 向量化 → **向量检索 + BM25 混合检索** → RRF 融合 → 大模型生成答案
- **交付形式**：命令行问答（`python src/qa.py`）

## 三、为什么不需要训练？为什么运行这么快？

这是 RAG 和传统「训练模型」的本质区别：

### 传统训练 vs RAG

| | 传统训练（微调） | RAG（本项目） |
|---|---|---|
| 做什么 | 调整模型参数，让模型"记住"知识 | 不改模型，把知识存进外部数据库 |
| 耗时 | 几小时到几天（需 GPU） | 索引构建约 10 秒，单次问答 2-5 秒 |
| 成本 | 高（算力 + 数据标注） | 低（API 按量付费，作业全流程 < 0.5 元） |
| 知识更新 | 需重新训练 | 只需更新文档并重建索引 |
| 答案可靠性 | 可能幻觉 | 基于检索到的原文生成，可标注来源 |

### 本项目实际在做什么

```
【离线阶段 — 只做一次，约 10 秒】
  13 篇 Markdown 文档
    → 按标题/段落切分为 105 个文本块
    → 调用 DashScope Embedding API 转为向量（预训练模型，无需训练）
    → 存入 FAISS 向量索引

【在线阶段 — 每次问答，约 2-5 秒】
  用户提问
    → 问题转向量（1 次 API 调用，< 1 秒）
    → 向量检索 Top-10 + BM25 关键词检索 Top-10（本地计算，< 0.1 秒）
    → RRF 融合排名，取 Top-4（互补语义匹配和精确关键词）
    → 将文本块 + 问题交给 qwen-plus 生成答案（1 次 API 调用，1-3 秒）
    → 输出答案 + 来源标注 + 免责声明
```

**快的原因**：
1. 没有训练过程，用的是现成的预训练模型（Embedding + LLM）
2. 知识库小（105 个向量），FAISS 检索毫秒级
3. 只需 2 次 API 调用（1 次 Embedding + 1 次 LLM）

## 六、运行方法

### 6.1 安装依赖

```bash
cd work10
pip install -r requirements.txt
```

### 6.2 配置 API Key

```powershell
$env:DASHSCOPE_API_KEY = "sk-你的密钥"
```

### 6.3 构建索引（首次运行，约 10 秒）

```bash
python src/build_index.py
```

> 提交包中已包含构建好的索引（`vectorstore/`），可直接跳到 6.4。

### 6.4 问答

```bash
# 单次提问
python src/qa.py --query "感冒有哪些主要症状"

# 仅检索（不调用 LLM，验证检索效果）
python src/qa.py --query "头痛" --retrieve-only

# 交互模式（输入 demo 运行 5 个预设问题）
python src/qa.py

# 一键演示（Windows）
run_demo.bat
```

## 八、技术要点总结

1. **文档分块**：按 Markdown 标题语义切分，保留章节信息作为元数据
2. **向量化**：DashScope text-embedding-v3（1024 维），L2 归一化后用内积检索
3. **混合检索**：FAISS 向量检索 + BM25 关键词检索（jieba 分词），RRF 融合排名
4. **相关性过滤**：向量最高分 < 0.25 时拒绝回答，避免胡编
5. **Prompt 工程**：System Prompt 约束模型只根据参考资料回答，附免责声明
6. **来源溯源**：每个答案标注引用文档和章节，可验证
