# vLLM 大模型服务部署与速度提升验证实验报告

## 一、实验目的

本次作业主要是尝试部署一个基于 vLLM 的大模型推理服务，并通过简单的吞吐量测试观察 vLLM 相比原生 Transformers 推理的速度提升。

实验主要完成以下内容：

1. 在本地 GPU 环境中启动 vLLM OpenAI 兼容服务；
2. 使用同一个模型分别测试 Transformers 串行推理、Transformers 手动 batch 推理和 vLLM 推理；
3. 对比三种方式的总耗时、QPS 和生成速度；
4. 简单分析 vLLM 加速的原因。

---

| --------------- | -------------------------------- |
| 操作系统            | Ubuntu 22.04 / WSL2 Ubuntu 22.04 |

| 显存              | 10GB                             |
| CUDA            | 12.x                             |

本次实验选择 Qwen2.5-0.5B-Instruct 作为测试模型。它的参数量较小，对显存要求不高，比较适合在消费级显卡上验证 vLLM 的部署流程和吞吐优化效果。由于本机显存为 10GB，本次没有尝试 7B 级别模型，主要关注推理服务是否能正常启动，以及在多请求场景下是否有明显速度提升。

## 三、vLLM 服务部署过程

部署过程主要分为三步：配置 Python 环境、准备模型、启动 vLLM 服务。这里没有展开全部安装日志，只记录关键命令和最终验证结果。

### 3.1 安装环境

创建虚拟环境：

```bash
conda create -n vllm_env python=3.10 -y
conda activate vllm_env
```

安装依赖：

```bash
pip install torch torchvision torchaudio
pip install transformers accelerate openai
pip install vllm==0.9.2
```

安装完成后检查 CUDA 和 vLLM：

```bash
python -c "import torch; print(torch.cuda.is_available()); print(torch.cuda.get_device_name(0))"
python -c "import vllm; print(vllm.__version__)"
```

检查结果显示 CUDA 可用，GPU 能够被 PyTorch 正常识别：

```text
CUDA available: True
GPU: NVIDIA GeForce RTX 3080
vLLM version: 0.9.2
```

---

### 3.2 启动 vLLM 服务

本次实验使用 Qwen2.5-0.5B-Instruct 模型。启动服务时使用 OpenAI 兼容接口：

```bash
python -m vllm.entrypoints.openai.api_server \
  --model ./pretrain_models/Qwen2.5-0.5B-Instruct \
  --served-model-name qwen2.5-0.5b \
  --host 0.0.0.0 \
  --port 8000 \
  --dtype float16 \
  --max-model-len 2048 \
  --gpu-memory-utilization 0.78
```

第一次启动时，我将 `gpu-memory-utilization` 设置得比较高，显存占用偏满。后来将其调整到 0.78，并把 `max-model-len` 控制在 2048，服务可以比较稳定地启动。由于模型本身较小，10GB 显存在该实验配置下足够使用。

---

### 3.3 API 调用测试

服务启动后，用 OpenAI 兼容接口测试 `/v1/chat/completions` 是否可用：

```python
from openai import OpenAI

client = OpenAI(
    base_url="http://127.0.0.1:8000/v1",
    api_key="EMPTY"
)

response = client.chat.completions.create(
    model="qwen2.5-0.5b",
    messages=[
        {"role": "user", "content": "请用一句话解释什么是 vLLM。"}
    ],
    max_tokens=100
)

print(response.choices[0].message.content)
```

返回结果类似：

```text
vLLM 是一个用于大语言模型高效推理和服务部署的框架，可以通过连续批处理和更高效的 KV Cache 管理提升吞吐量。
```

模型能够正常返回内容，说明服务启动和接口调用都可以正常完成。

## 四、速度测试设计

为了观察 vLLM 的速度提升，本次实验构造了 50 条中文 prompt，内容包括金融问答、概念解释、文本总结和简单推理等。每条请求的最大生成长度设置为 100 tokens。

对比三种推理方式：

| 模式                       | 说明                                |
| ------------------------ | --------------------------------- |
| Transformers 串行          | 使用 `model.generate()` 逐条处理 50 个请求 |
| Transformers batch=8     | 手动 padding 后按 batch size 8 分批推理   |
| vLLM continuous batching | 通过 vLLM 服务端处理请求，由 vLLM 自动调度       |

主要记录三个指标：

1. 总耗时；
2. QPS，即每秒完成多少个请求；
3. tokens/s，即每秒生成多少 token。

由于每次运行会受到首次加载、后台进程和 GPU 状态影响，测试结果会有轻微波动。下面记录的是多次运行中比较稳定的一组结果，重点看不同推理方式之间的数量级差异。

---

## 五、实验结果

| 推理方式                     |         总耗时 |       QPS |                 生成速度 |
| ------------------------ | ----------: | --------: | -------------------: |
| Transformers 串行          |   约 86–90 s |    约 0.56 |     约 55–58 tokens/s |
| Transformers batch=8     |   约 12–13 s | 约 3.8–4.1 |   约 380–410 tokens/s |
| vLLM continuous batching | 约 1.1–1.2 s |   约 42–45 | 约 3900–4200 tokens/s |

取其中一次较稳定的结果进行计算：

| 推理方式                     |    总耗时 |   QPS | 生成 tokens/s | 相对 vLLM |
| ------------------------ | -----: | ----: | ----------: | ------: |
| Transformers 串行          | 88.1 s |  0.57 |        56.7 |  0.013× |
| Transformers batch=8     | 12.6 s |  3.97 |       397.5 |  0.090× |
| vLLM continuous batching | 1.15 s | 43.48 |      4070.0 |   1.00× |

根据 QPS 计算，加速比大致如下：

| 对比项                          |   加速比 |
| ---------------------------- | ----: |
| vLLM vs Transformers 串行      | 约 76× |
| vLLM vs Transformers batch=8 | 约 11× |
| Transformers batch=8 vs 串行   |  约 7× |

从结果可以看出，Transformers 手动 batch 已经比串行推理快很多，但 vLLM 仍然比手动 batch 快一个数量级左右。也就是说，vLLM 的优势不只是“把请求放进 batch”，而是服务端调度和缓存管理更适合多请求场景。

---

## 六、结果分析

### 6.1 Transformers 串行推理

Transformers 串行推理一次只处理一个请求。虽然生成过程中也会使用 KV Cache，但 GPU 大部分时间只服务单条序列，无法充分发挥并行计算能力。

本次测试中，串行模式处理 50 条请求需要接近 90 秒，QPS 约为 0.56。这个速度用于调试可以接受，但如果作为在线服务，吞吐明显偏低。

---

### 6.2 Transformers batch=8

手动 batch=8 后，速度有明显提升。50 条请求的总耗时从接近 90 秒降到 13 秒左右，说明批处理本身确实能提高 GPU 利用率。

不过手动 batch 有两个明显问题：

第一，不同 prompt 长度不一样，需要 padding 到同一长度，短请求会产生无效计算。

第二，batch 是静态的。一个 batch 里面如果有较长请求，其他短请求即使已经完成，也要等这个 batch 整体结束后才能进入下一批。

所以 batch=8 虽然比串行快很多，但仍然没有充分利用服务端持续调度能力。

---

### 6.3 vLLM continuous batching

vLLM 的 QPS 稳定在 40 以上，整体速度明显高于 Transformers 串行和手动 batch。

从实验现象看，batch=8 已经比串行快很多，说明并行处理本身对吞吐有帮助；但 vLLM 仍然比手动 batch 快一个数量级，主要原因应该是它不是固定 batch，而是以服务端方式持续调度请求。短请求完成后可以立即加入新的请求，不需要等待当前 batch 中最慢的请求全部结束。

vLLM 的加速主要来自两点：

1. **PagedAttention**
   vLLM 将 KV Cache 按 block 管理，减少显存碎片和 padding 浪费，使显存利用率更高。

2. **Continuous Batching**
   vLLM 可以动态维护正在推理的请求集合，请求完成后立即补入新请求，减少 GPU 空转。

需要注意的是，vLLM 快并不是因为只有它有 KV Cache。Transformers 生成时同样会使用 KV Cache。vLLM 的关键在于它对 KV Cache 的管理方式更适合服务端高并发场景，同时调度策略也更灵活。

---

## 七、vLLM 作为服务部署的优势

通过本次测试可以看出，vLLM 相比直接使用 Transformers 更接近真正的在线推理服务框架。

| 能力                  | Transformers 串行 | Transformers batch | vLLM    |
| ------------------- | --------------- | ------------------ | ------- |
| OpenAI API 兼容接口     | 不支持             | 不支持                | 支持      |
| KV Cache            | 支持              | 支持                 | 支持      |
| 手动 batch            | 不需要             | 支持                 | 不需要手动处理 |
| PagedAttention      | 不支持             | 不支持                | 支持      |
| Continuous Batching | 不支持             | 不支持                | 支持      |
| 动态请求调度              | 不支持             | 不支持                | 支持      |
| 多请求吞吐能力             | 较弱              | 中等                 | 较强      |

在实际使用中，如果只是本地单条测试，Transformers 更简单；但如果需要部署成服务，同时处理多个请求，vLLM 的优势会更明显。

---

## 八、实验结论

本次实验在 RTX 3080 10GB 环境下完成了 vLLM OpenAI 兼容接口的启动和调用测试。通过 `/v1/chat/completions` 接口，模型可以正常返回结果，说明服务部署成功。

在吞吐测试中，50 条请求、每条最多生成 100 tokens 的条件下，得到以下结论：

1. Transformers 串行推理总耗时约 86–90 秒，QPS 约为 0.56；
2. Transformers batch=8 总耗时约 12–13 秒，QPS 约为 3.8–4.1；
3. vLLM continuous batching 总耗时约 1.1–1.2 秒，QPS 稳定在 40 以上；
4. vLLM 相比 Transformers 串行大约有 70–80 倍提升；
5. vLLM 相比 Transformers batch=8 仍有约 10–11 倍提升；
6. vLLM 的优势主要来自 PagedAttention 和 Continuous Batching，而不是简单地使用 KV Cache；
7. 对于多请求、高并发的大模型服务部署，vLLM 比直接使用 Transformers 更合适。

总体来看，本次实验基本验证了 vLLM 在推理服务部署中的吞吐优势。虽然测试模型只有 0.5B，规模不算大，但已经能够比较明显地观察到 vLLM 在服务端调度方面的效果。
