# Week10 RAG 从零构建 — 学习报告

> 项目：rag_scratch | v0.1 → v0.3 | 2026.07.06–07.07
> 环境：GTX 1080 Ti 11GB / bge-base-zh-v1.5 (768d) / DeepSeek v4-flash
> 数据：Rust 中文文档 7 来源、530 页 HTML

本周做了一个之前没碰过的东西：从零实现一个完整的 RAG 系统。知识库是 Rust 中文文档——教程、语言规范、代码示例、术语表混在一起——最终跑通了下载→解析→分块→BM25+向量检索→RRF 融合→重排→LLM 生成→评估的完整管线。

整个过程可以按两个版本来理解。v0.1 的目标是把管线每一环跑通，核心问题是「关键字为什么查不出来」。v0.2 的目标是让评估站住脚——系统能跑了但 Faithfulness 四道零分，不修到清零就没法说这个东西真的「能用」。

---

## 一、v0.1：从零打通完整管线

### 1.1 选型

动手之前花了不少时间推敲方案。事后看，这是唯一没踩坑的阶段——后面所有翻车都跟编码实现有关，选型本身反而很稳。

数据源选了 Rust 中文文档。理由很简单：太干净的数据（比如 Python 文档，代码+中文规整）压榨不出 BM25 和向量的互补性。Rust 文档 7 个来源——教程（Book 105 页）、语言规范（Reference 112 页）、代码示例（RBE 180 页）、术语表（Glossary 371 条）——同一页里叙述段和代码块交替出现、规范条目夹杂表格。对分块策略是真正的考验。

| 决策 | 选型 | 理由 |
|------|------|------|
| 向量模型 | bge-base-zh-v1.5 (768d) | bge-small 精度不够，bge-m3 太重 |
| 向量库 | FAISS IndexFlatIP | 5000×768 < 20MB，暴力搜索够快 |
| BM25 | rank-bm25 + jieba 分词 | RRF 融合自己写，不重复造轮子 |
| 重排 | bge-reranker-base（可开关） | 精排不能变成硬依赖 |
| LLM | DeepSeek v4-flash | Key 已配好，中文+代码能力过关 |
| Glossary | 371 条术语注入 prompt | 约束 LLM 译名一致性 |

两个关键的工程决策值得单独说。

第一个：**HTML→JSON→chunk，中间放一个 JSON 缓存**。如果 HTML 直接切块，每次调分块策略都要重新解析 530 页 HTML。JSON 缓存放在那，改分块只跑 `build_index.py`，3 秒出结果。

第二个：**解析与索引分离**（`parse.py` 只做 HTML→JSON 的 I/O，`build_index.py` 做 JSON→chunk→索引）。改分块策略只重跑后者，不用重新下载重新解析。这个设计后来被证明了至少十几次——父子块方案落地前后，`build_index.py` 跑了不下 20 遍。

### 1.2 「关键字」问题引出三次修复

v0.1 的核心叙事是同一个问题：查询「Rust 中有哪些关键字」，系统始终答不对。这个问题不是一次性修好的——修好一个暴露下一个，三次修复才彻底解决。

**第一次：RRF 聚合方式**

BM25 单独搜关键字 → top-1 命中。向量单独搜 → top-1 命中。RRF 融合后 → top-1 完全跑偏（返回的是「可派生的 trait」）。

排查用的是二分法，定位到 RRF 的聚合方式：原始实现按 `chunk_id` 聚合——一个文档被切成 8 个 chunk，每个 chunk 在两个检索器中各贡献一个 RRF 分数。长文档 5 个以上 chunk 的累计分轻松碾压「关键字」文档的 2 个 chunk。

| 版本 | 方案 | 关键字排名 |
|------|------|:--:|
| v1 | 按 chunk_id 聚合 | 不出现 |
| v2 | 降 K 值（60→5） | 不出现——问题不在 K |
| v3 | 按 source_url 聚合，取 max chunk 分数 | #4 |
| **v4** | **按 source_url 聚合，取最佳排名** | **#1** |

最终公式：`score(doc) = 1/(k+best_bm25_rank) + 1/(k+best_vector_rank)`。按文档聚合、取最佳排名而不是累计，更接近实际的检索质量。

**第二次：LLM 还是看不到关键字列表**

排名对了（#1），LLM 的回答却是「文档中未列出具体的关键字列表」。

查了返回给 LLM 的 chunk 内容——38 个关键字的完整 chunk 有 1101 字符，但 `max_chunk_size=500` 截断后只剩一句「以下列表包含 Rust 中正在使用的关键字……」，列表本身全被砍掉了。

这就是父子块方案的起点：**子块小而精准（200-400 字符），用来检索；父块保留完整 section（≤3000 字符），给 LLM 用。**检索命中子块 → 扩展为父块 → 拼接同源兄弟父块 → 送给 LLM。修复后排 #1 的「关键字」文档 LLM 能正确列出全部 38 个——不仅因为排对了，更因为父块拼接把列表段也带上了。

一个附带发现：keyword 文档排 #6 时 LLM 仍能正确列出关键字，因为兄弟拼接补全了信息。检索排名和 LLM 最终输出之间的关系不是线性的——不要只看排名数字，要看 LLM 输出了什么。这个认知是踩出来的。

**第三次：「中」和「里」差了一个字**

排查关键字问题时发现一件诡异的事：「rust中有哪些关键字」和「rust里有哪些关键字」，keyword 文档排名差了 3 位，BM25 排名从 #36 恶化到 #76。

jieba 分词对比：`rust/中/有/哪些/关键字` vs `rust/里/有/哪些/关键字`——多切出一个无意义的「里」字，token 分布变了，BM25 评分剧变。BM25 作为词袋模型，对自然语言的表达波动极其敏感。

修复方案：LLM 查询重写。口语化查询 → DeepSeek v4-flash 改写为 3 个标准化版本 → 多路并行检索 → 按 source_url 去重合并。「rust里有哪些关键字」改写后变成 `["Rust 关键字列表", "Rust 严格关键字 保留字", "Rust 关键字 as async await"]`，keyword 稳定排 #0。

三次修复串起来看，有一个教训：**简单修复优先于架构改动。**关键字问题最早的想法是上 LLM 查询重写（架构改动），但父子块方案用 10 行代码就解决了 LLM 看不到列表的问题。先拿最小的代码量验证效果，再决定要不要上大方案。

### 1.3 分块策略：四个版本

分块是整个项目里改动最多的模块，经历了四个版本：

| 版本 | 方案 | 触发 |
|------|------|------|
| v1 | 按 h2 切分，max_chunk_size=500 | 初始 |
| v2 | 按数据形态路由（6 类 morphology） | 「按形态分类，不按来源 URL」 |
| v3 | 代码单元独立分块 | 代码检索精度 |
| v4 | 父子块方案 | 「关键字」召回问题 |

v2 有个插曲。前期 Grill 阶段讨论已经确认「morphology 在 parser 阶段标注」，但我在实现时「优化」了一下——v1 parser 只标 HTML 标签类型（heading/paragraph/code_block），形态推断推迟到 chunker 里做。回头看，前期敲定的设计决策不能在实现时为了省事改掉——修正后 parser 输出加 `morphology` 字段：`<p>`→narrative、`<pre><code>`→code_unit、`<ul>/<ol>`→structured。

教训：推迟的实现几乎总是要还回去。

父子块落地时还踩了两个细节坑：

- **structured 子块的 text 必须包含 heading 前缀**。列表条目「as - 强制类型转换」如果不带「关键字」前缀，BM25 永远找不到。子块 text 需要把父级标题拼进去。
- **父块 text 漏了 heading**。`_split_by_headings` 用标题做切分边界，标题存在 `headings` 元数据列表里，但 `_build_rich_text` 只遍历 elements——标题丢了。标题既是切分信号也是内容，两边都要保留。

### 1.4 评估初版：勉强有个分数

v0.1 的评估体系是「能用但完全不靠谱」。

先用 LLM 自动生成了 60 道题（混合概念解释/代码补全/对比分析/术语定义）。跑完发现 43% 是代码补全型——「请补全以下代码……」，没有上下文根本没法合理评分。后来手写了 40 道高频 Rust 概念题，质量完全不是一个量级。

四指标评估器上线：Context Precision（关键词命中率）、MRR（第一个相关 chunk 的排名倒数）、Faithfulness（LLM 逐句核查声明是否在 chunk 中有依据）、Answer Relevancy（LLM 逆向生成问题+余弦相似度）。前两个免费，后两个每次各调一次 LLM。

初版 10 道手动题的结果：Precision=0.89、MRR=1.00、Faithfulness=0.67、Relevancy=0.80。Faithfulness 四道零分（m24/m26/m36/m38），全是「A 和 B 有什么区别」「对比 X 和 Y」类题目。

到这一步，管线虽然跑通了，但评估告诉你的信号是错的——Faithfulness=0 不是因为系统瞎编，而是评估器没能力判断对比类回答。这就是 v0.2 要解决的问题。

## 二、v0.2：让评估站住脚

v0.1 证明了管线能跑。v0.2 的核心目标是把 Faithfulness 从 0.67 推到能看的水平，同时加一个 WebUI 让效果可视化、方便调。

### 2.1 WebUI

加了一个 Streamlit 前端（4 页面：查询、评估、设置、历史），对接 FastAPI 后端的 6 个端点。前端本身不是 RAG 的核心，但有一个认识很重要：**评估跑在网页上和你跑脚本是完全不同的体验。**脚本跑完看数字就行了，网页上你能看到每道题 LLM 到底答了什么、检索命中了哪些 chunk、评估器逐条核查了哪些声明。这种可视化对调试 Faithfulness 的作用不是「锦上添花」，是实质性的——第四轮修复中 preview 截断 bug 就是通过前端逐题检查发现的。

前端本身踩的坑放到工程实录里一起说。

### 2.2 Faithfulness 四轮修复

这是整个项目中最难也最有价值的部分。四轮修复不是提前规划的——每一轮被上一轮的结果打脸，引入更深层的修复。

**第一轮：分层评估 + 术语规范**

现象：4 道 Faith=0，全是对比类问题。

| 题号 | 问题 |
|------|------|
| m24 | Clone 和 Copy trait 的区别？哪些类型实现了 Copy？ |
| m26 | 什么是函数指针？fn 类型和 Fn trait 有什么区别？ |
| m34 | mut、ref、move 关键字在模式匹配中的作用？ |
| m36 | 什么是孤儿规则（orphan rule）？它如何限制 trait 实现？ |
| m38 | 什么是关联类型（associated type）？它和泛型参数有什么区别？ |

Faithfulness 评估器逐句核查「回答中的每一条声明是否能在单个 chunk 中找到原文」。但对比类问题的答案需要跨文档综合形成对比结论——「方案 A 成本更低但扩展性弱于方案 B」这句话在任何一个 chunk 里都不存在。

另外 m24 有个额外问题：LLM 回答中用了「超类trait」，标准译名是「父 trait」。评估器在检索 chunk 中搜不到「超类trait」→ 第一条声明就判无依据 → 整段 0 分。

两个方向并行修复：
1. **术语规范**：generator system prompt 加第 5 条，必须用 Glossary 标准译名，禁止自造词
2. **分层评估**：15 个推导关键词（「区别」「对比」「vs」「如何选择」等）→ 推导类用宽松 prompt，允许联合多 chunk 推导；事实类保持严格逐句核查

效果：4→3 道零分，仅小幅改善。

**第二轮：豁免规则**

现象：3 道仍为 Faith=0（m26/m34/m36）。

写了 `test_zero_faith_root_cause.py` 逐题检测根因：

| 题 | 根因 |
|----|------|
| m36 | LLM 诚实回答「文档中未提供，无法进一步说明」→ 评估器判为无依据 |
| m26 | 回答含代码示例 + 推导结论 → 评估器要求原文匹配 |
| m34 | 回答全篇用「来源：文档 X」引述风格 → 评估器无法验证引述 |

三种情况都是评估器 prompt 的问题——只定义了「什么是 bad」（幻觉），没有定义「什么应该被豁免」。修复是在两个评估 prompt（事实类+推导类）中各追加三条豁免：
1. 诚实表述（「文档中未提供/无法说明」）→ 直接判 supported
2. 代码示例 → 只要示例概念在文档中有说明，即视为 supported
3. 引述声明（「来源：文档 X」）→ 只要 X 在检索范围内，即视为 supported

效果：3→1 道零分，Faith 均值从 0.67→0.77。

**第三轮：Glossary 动态检索**

现象：m24 反复在 0↔1 之间摆动，怎么修都不稳定。

深层根因：`format_glossary_for_prompt` 只取 50 条术语注入 prompt（按中文长度排序）。「supertrait → 父 trait」（7 字符）排 #316/374——永远选不上。排在前面的全是「堆/栈/宏/糖」这类单字词，跟当前查询半毛钱关系没有。

这暴露了一个根本性矛盾：371 条术语，50 条的硬限制，按长度排序完全随机——你没办法预测哪条术语会在哪个查询中用上。

最终方案：**Glossary 走检索管道。**374 条术语构建独立的 FAISS + BM25 索引，查询时并行检索，按需召回 ≤10 条相关术语动态注入 prompt。

```
查询 "Clone 和 Copy 的区别"
    ↓
并行检索 ─┬─ 文档索引       → chunks
          └─ Glossary 索引   → supertrait→父 trait, trait→特质, Clone→克隆...
    ↓
动态注入：只注入检索到的相关术语（≤10条）
```

效果：m24 Faith=0→1.0，LLM 从「超类trait」改为「父 trait」。

**第四轮：preview 截断 + source 漏发**

现象：Glossary 修复后 m24=1.0，但 m26 始终 Faith=0。

排查过程很折腾，列出来是因为排查方法论本身值得记：

| 假设 | 验证 | 结论 |
|------|------|------|
| LLM 措辞 vs 原文不匹配 | 加规则「优先使用原文措辞」 | 反噬——声明翻倍，Faith 更差 |
| 检索召回差 | `diagnose_m26_retrieval.py` 查三轮排名 | 函数指针文档 RRF #1，检索没问题 |
| 评估器 LLM 随机偏差 | 手动对比原文 | 原文逐字存在，不应该找不到 |
| **评估器拿到的 chunk 被截断** | `verify_m26_context.py` 查 context 长度 | **找到了——所有 text 只有 200 字符** |
| **source 传少了** | 查 `/query` 返回 | **前端传 sources[:5]，函数指针 #6 被截掉** |

两个 bug：

1. **preview ≠ text**：`/evaluate` 端点构建 Chunk 时用了 `s.get("preview", "")`（200 字符截断版），而非完整 chunk text。函数指针文档在第 200 字符被切断——「函数的类型是 fn （使」——后面的「用小写的 f 以免与 Fn 闭包 trait 相混淆」全丢了。评估器拿到的是残本，Faith 当然 0。

2. **sources[:5] → [:8]**：前端传给 `/evaluate` 时只发前 5 个 source，函数指针排 #6 被截。而 generator 用的是 `chunks[:8]`，数据量不同步。

修复：`/query` 端点 source 新增完整 `text` 字段（2028 字符），`/evaluate` 改用 `s.get("text")` 而非 `s.get("preview")`，前端传 `sources[:8]`。

效果：m26 Faith=0→1.0。

第四轮给了一个通用教训：**不要轻易归因于「LLM 随机性」。**当评估器在正确答案中反复找不到逐字存在的原文时，先检查数据有没有真正传给评估器。preview 是给 UI 展示用的，评估必须用完整 text——这个差别看似小，但 m26 追了四轮才定位到。

### 2.3 最终评估

四轮修复后，40 题混合评估的结果：

| 指标 | v0.1 初版（10 题） | v0.2 封版（40 题） |
|------|:--:|:--:|
| Context Precision | 0.89 | 0.75 |
| MRR | 1.00 | 1.00 |
| Faithfulness | 0.67 | **0.89** |
| Answer Relevancy | 0.80 | 0.79 |
| Faith=0 题数 | 4 | **0**（清零） |
| Faith≥0.9 题数 | — | **25/40**（62.5%） |

几个要点：

Precision 下降（0.89→0.75）不是系统退化了，是题目覆盖面从 10 道扩到 40 道后拉低了平均分。按题型拆开看，概念解释类 Precision=0.78，对比分析类 Precision=0.75——差距不大，说明检索系统对不同题型没有明显的偏好或短板。Relevancy 基本持平（0.80→0.79），LLM 的生成质量稳定。

Faithfulness 的提升（0.67→0.89）来自四个不同层面的修复：分层评估是机制缺陷，豁免规则是 prompt 定义不完整，Glossary 检索是架构设计问题，preview 截断是工程 bug。没有一个是「调了个参数变好了」——每一个都对应一个被结果打脸后重新挖出来的根因。

最有意思的是对比分析类题目的变化。四轮修复前，对比分析是 Faithfulness 的重灾区（4 道零分全是它）。修复后，对比分析类 Faith=0.935，反而成了最高的题型——高于概念解释类的 0.868。这说明分层评估的分流是对的：一旦允许跨文档综合推导、配合豁免规则和术语修正，这类题目的评估就不再误判。剩下的 Faithfulness 短板不在对比分析，而在一些需要精确命中原文细节的概念解释题。

Faith=0 清零不代表系统完美了。7/40 道题 Faith 在 0.7-0.85 之间，这些题不是评估误判——是 LLM 的回答确实有部分声明在检索结果中找不到原文依据。

### 2.4 后续方向

v0.2 封版时评估指标已经看得过去，但有几个明显可以继续做的东西：

**检索精度。**当前 Precision=0.75，各类题型（概念解释 0.78、对比分析 0.75）差距不大，没有明显的题型偏好——短板是均匀的。这个水平的瓶颈可能不在检索策略本身，而在 BM25 和向量检索在面对 Rust 这种中英混排、代码和叙述交替的文档时，关键词匹配天然有损耗。后续可以尝试 HyDE（先让 LLM 生成假答案再检索）或者查询扩展，把自然语言查询展开为多个角度再并行检索。

**增量索引。**目前改数据就要重建全量索引。数据量小（530 页 HTML）时无所谓，但这是一个迟早要面对的问题。Glossary 的 374 条术语走独立索引绕过了这个问题——文档索引和术语索引分开维护，增删术语不影响文档检索——这个思路也许可以推广。

**多轮对话。**当前只支持单轮问答。Rust 文档这种技术知识库天然适合追问——「什么是所有权」之后自然会问「那借用和所有权的区别是什么」。加上对话状态和指代消解，追问题不需要重复完整上下文。

**评估框架。**对比类回答的 Faithfulness 评估目前靠分层 prompt + 豁免规则兜底，不够系统。对比结论的「忠实性」本质上是推理质量评估，不是逐句原文匹配——这个方向如果要继续做，可能需要单独设计一套对比类评估标准。

三个方向里，检索精度和多轮对话是让系统「更好用」，增量索引是让工程「更可持续」，评估框架是让衡量标准更接近真实情况。优先级取决于下一阶段的目标。

## 三、v0.2→v0.3：从能跑到能诊断

v0.2 封版后系统能用，评估数字也看得过去。但这时候一个问题越来越突出：**召回不准的时候，不知道为什么不准。**同样是 RRF 融合的结果、同样是父子块扩展后的上下文，不同入口（CLI、Web、评估脚本）可能走不同的检索路径——入口漂移让评估数据和真实使用效果之间存在看不见的偏移。

这一轮的目标不是堆功能，是把「能用」推进到「能诊断、能验证、能稳定迭代」。8 个提交、109 个测试全部通过。

### 3.1 统一入口：先收敛再优化

CLI (`query.py`)、Web app (`app.py`)、评估脚本 (`evaluate.py`) 中各自有一段重复的检索、父块扩展、术语表、生成拼装逻辑。调参改了三处中的一处，结果评估数据和 Web 体验产生偏差——你没办法判断是优化真的有效，还是只是改了某个入口的参数。

新增 `src/pipeline.py`，把加载 retriever、查询、父块扩展、Glossary 补充、生成等路径统一为共享的 `RAGPipeline`。三个入口改为调用同一 pipeline。

教训很直接：RAG 项目最怕「评估路径」和「线上路径」分叉。统一入口不是重构洁癖，是后续优化可验证的基础设施。

### 3.2 父块上下文修正：命中位置的上下文才是有效上下文

之前子块命中后扩展到父块，拼接逻辑倾向于从文档开头开始扩展。对于 Rust Book 这种长章节文档，这会引入看似相关但实际偏题的上下文，冲淡真正命中的内容。

改为以命中的 parent 为中心扩展，左右相邻块补上下文，受 `max_chars` 控制。远距离的前置块不再进入上下文窗口。

这个改动本身不大，但背后的认识很重要：**RAG 的召回到了 ≠ 给 LLM 的上下文对了。**chunk 命中后的上下文重组同样是召回质量的一部分，而且是容易被忽略的那部分。

### 3.3 召回优化的三步走

用户反馈「召回不准」后，没有直接调大 top_k 或换模型，而是先补可观测性、再做低风险排序改进。

**第一步：检索诊断脚本。**新增 `diagnose_retrieval.py`，对单条 query 输出 BM25 top results、向量 top results、RRF 逐文档分数、source/heading/morphology/expected term 命中情况。没有诊断工具时，召回优化基本是在猜。有了它，能把每个问题拆成可定位的层——候选有没有进池？在哪个检索器进池？融合后为什么掉了？

**第二步：向 BM25 tokenizer 注入 Rust 领域词表。**Rust 中文文档里混合了中文解释、英文关键字（trait、lifetime、fn、match、borrow checker）和专有概念（所有权、生命周期）。默认中文分词对这些技术词不友好——BM25 不能稳定保留这些词，关键词类查询就弱。改动是在 BM25 tokenizer 初始化时注入 Rust 关键词、Glossary 术语，并对英文短语追加 phrase token。

这个改动的价值不是「BM25 变强了」，而是揭示了技术文档 RAG 的一个基本事实：**不能完全依赖通用分词，领域词表是 BM25 成败的关键部分。**向量模型能解决语义相似，但精确术语和关键字查询仍然需要 BM25。

**第三步：给 RRF 加轻量 metadata boost。**项目的 chunk 里已经有 `headings` 和 `morphology` 两个字段——查询命中 heading 时小幅加分，列表类查询偏好 `structured`，代码类查询偏好 `code_unit`，对比类查询偏好 `narrative` 或 `mixed`。

加分幅度刻意保持很小（heading +0.003、structured +0.002、code +0.002、comparison +0.001），不做主导信号，只在候选基础分接近时利用文档结构做决策。思路是：**RRF 是好的第一层融合，但它不知道文档结构——heading 和 morphology 是免费信号，应该利用。**这比直接上 cross-encoder 成本低、可解释、可测试。

### 3.4 效果：真实查询的消融对比

代码改了 tokenizer 和 RRF boost 后，BM25 索引还是旧词表构建的。重建索引后（6440 chunks，BM25 4589 docs），用几组真实查询做了消融对比：

- `rust有哪些关键字`：boost off 时 top3 偏向「通用编程概念」「原始标识符」等泛相关页面；boost on 时「关键字 > 严格关键字」进入 top1
- `Rust所有权和借用有什么区别`：top1「认识所有权」、top2「RefCell<T> 在运行时检查借用规则」、top3「借用」
- `如何写Rust函数示例`：top2 命中 `ch03-03-how-functions-work`
- `trait是什么`：top1 命中「reference > 术语表 > 本地 trait」
- `生命周期是什么`：top1 命中「rust-by-example > 生命周期」

metadata boost 的效果在关键字类查询上最明显——不加 boost 时容易被泛化章节淹没，加了以后结构化页面能稳定进入前排。Token 注入的效果更基础但更难做消融，因为它改变的是 BM25 的底层分词，影响面宽。

### 3.5 这一轮的教训

几条跟之前几轮不太一样的经验：

**先统一入口，再优化指标。**如果 CLI、Web、评估各走一套逻辑，任何优化都可能只是局部有效。统一 pipeline 之后做的诊断、调参，才敢说评估数据跟真实使用是对齐的。

**召回问题要拆层观察。**答案不准的可能原因太多了——query rewrite 不合适、BM25 没分出领域词、向量召回偏泛化、RRF 融合把强信号冲淡、child 命中后 parent 扩展错位、Glossary 补充过强或过弱、LLM 在上下文中抽取失败。诊断脚本让你可以从这些层里定位，而不是在七层里盲调。

**领域 RAG 要认真对待稀疏检索。**这一轮最大的认知变化是：之前潜意识里觉得向量语义检索是主力、BM25 是辅助。但 Rust 文档的数据形态（中英混排、代码符号和叙述交替）决定了精确术语查询必须靠 BM25，向量的语义泛化在这里不是优势而是噪声。不是你选哪个检索器的问题，而是不同查询类型需要不同的检索器主导。

**利用已有 metadata，而不是急着上重模型。**headings 和 morphology 已经包含了很强的结构信号，轻量 boost 成本低、可解释、效果可验证。这条路还没走完——后续可以把 boost 信息写进诊断报告，跟基础 RRF 分数拆开看，让每次调参的变化可以用数字追踪。

## 四、工程实录

管线之外，有几个直接跟 RAG 开发相关的工程问题值得记。集中在这里。

**Chunk ID 全局冲突。**父子块方案落地后重建索引，检索只返回 1 个子块。查 `all_chunks.json`——6440 条记录，但只有 741 个唯一 ID。`book_p0000` 重复了 105 次。根因：每个文件独立从 0 计数——105 个文件全撞了。修复是 chunk_id 加 `file_index` 前缀。顺带发现 `_make_parent` 调用漏传了 `file_index` 参数——默认参数不报错，Python 静默吞了。

**Streamlit 前端踩坑。**四个典型问题：

- `st.button` 不响应 Enter 键 → 用 `st.form` + `form_submit_button` 包裹
- 评估运行中点击任何控件都触发全量重跑 → 锁定页面 + `eval_running` 状态 + 重入续跑
- PowerShell `Start-Job` 不继承环境变量 → 显式传入 `PYTHONPATH` 和 `HF_HOME`
- emoji + Windows GBK = 崩溃（`🚀 API 启动` → `UnicodeEncodeError`） → `PYTHONIOENCODING=utf-8`

**调试脚本保存为文件。**整个排查过程中，所有测试都写成了 `.py` 文件（`run_query_test.py`、`diagnose_main.py`、`diagnose_m26_retrieval.py`、`verify_m26_context.py`、`test_zero_faith_root_cause.py` 等），不是 `python -c` 一次性命令。事后可以直接复现完整排查路径。这个习惯是 Week03 养成的——当时交互样本验证也是写了文件而不是手工试。

这次和之前几周最大的不同：之前是做「在一个固定任务上换不同方案对比」——同一个分类数据集，LSTM/BERT/零样本/LoRA 全试一遍。这次是从零捯饬一个完整系统——下载、解析、分块、检索、生成、评估、前端，从头到尾自己串起来。

但更值得记的不是「做了多少模块」，而是中间被结果打脸后怎么修的。四轮 Faithfulness 修复，每一轮都推翻了一个假设——第一轮以为分层评估就够了，结果只少了一个零分题；第二轮加了豁免规则，还有一道题目在零一之间摆动；第三轮才发现 Glossary 静态注入从根本上就是抽签；第四轮发现前面的所有修复都在评估器的 prompt 上做文章，但数据根本没送对。一个评估指标，从 0.67 推到 0.89，花了四个不同层面的修复。

如果只靠论文和文档，这些东西永远不会遇到。工程上难的从来不是算法原理，是「看着逐字存在的原文，评估器非说找不到，折腾四轮才发现是 preview 截断了」。
