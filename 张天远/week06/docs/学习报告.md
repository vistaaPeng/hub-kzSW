# 四周文本分类学习报告

从 Week03 到 Week06，我一直在做文本分类。同一件事，换了五套方案——LSTM、BERT、自建 GPT、LLM 零样本、LLM SFT+LoRA。

写这份报告不是罗列实验结果，是把我对"模型"这件事的理解怎么一步步变的，记下来。

## 一、Week03：从 BiLSTM 到 BERT——一个模型改了五次

cnews 数据集，中文新闻 10 分类，5 万训练 / 5 千验证 / 1 万测试。但我不是只写了一个 LSTM——我迭代了五个版本，从最简陋的纯 BiLSTM 一路改到 BERT。回头看，迭代过程学到的东西比任何一个单模型都多。

### 阶段 1：纯 BiLSTM——11.8M 参数，93.24%

第一版：Embedding(300) → BiLSTM(2×512) → 全连接分类。

一开始试 BATCH=256、HIDDEN=512、LAYERS=3——直接卡死。GPU 不报 OOM 也没反应，查了半天才找到：显存打满了（10.8/11 GB）。降到 BATCH=128、LAYERS=2，峰值 ~4 GB，才稳定跑起来。

**1080 Ti 第一个坑**：Pascal 架构没有 Tensor Cores，AMP 混合精度加了不加速，反而一堆兼容问题。最后只能纯 FP32。

**Windows 第二个坑**：DataLoader 的 `num_workers>0` 用 spawn 开多进程，直接死锁。改成 `num_workers=0` 解决。

训练完，一测：93.24%——数字看着还行。但手输几条样本，立刻露馅：

| 样本 | 预测 | 实际 |
|------|:----:|:----:|
| "中国女排横扫巴西夺世界杯" | 家居 ✗ | 体育 |
| "发改委大佬落马" | 家居 ✗ | 时政 |
| "乒乓球世界杯新华社长文" | 科技 ✗ | 体育 |

我第一反应是完全懵的——"女排"和"家居"有什么关系？后来仔细查才发现根因：**字符级模型学不到多字固定搭配。"女排"被切成 "女"+"排"，"中国"被切成 "中"+"国"。模型没在学语义，它在学"文体格式"——训练集里体育新闻全是短标题，家居新闻才用长句描述，模型直接把"长句 = 家居"当规律了。** 高准确率 ≠ 理解语义——这个教训是手输样本输出来的，不是看指标看出来的。

### 阶段 2：BiLSTM + Attention——94.51%

给 LSTM 输出加了 Additive Attention。Embedding 和 LSTM 权重从阶段 1 迁移过来，只新训 Attention 层。

结果：最佳准确率出现在 **epoch 1（91.01%）**，之后一路降——**迁移学习的学习率没调好**，新层的梯度把老权重冲散了。

迁移的参数和新层，学习率得分开设。后来 Week06 的分层学习率，其实就是同一个道理，只不过当时还没意识到。

### 阶段 3：CNN + BiLSTM + Attention——95.33%

BiLSTM 路线的最好成绩。加了多尺度 CNN（kernel=2,3,4，残差连接），参数量 13.7M，验证 F1=0.9534，测试准确率 **95.33%**。

CNN 多尺度卷积等于一个"软分词器"——自动学出"女排""横扫"这些 n-gram 搭配，残差连接保证字级特征不丢。测试集比纯 BiLSTM 高了 2 个点。

**但交互样本反而更差。** CNN 太擅长拟合训练集分布了，对"新华社格式 = 时政"这种文体偏置更自信了。测试集分数更高，实际泛化更差。这是我第一次见到**测试集分数和真实泛化能力不是一回事**——之前看论文知道这个说法，自己遇到是另一回事。

### 阶段 4：+ Jieba 分词——94.28%（不升反降）

既然字符级有问题，换 Jieba 词级试试。词表 3 万，其他不变。

结果：94.28%，比字级 CNN+Attn 还低。Jieba 的分词规则是固定的，"女排"要能在词表里才有，人名这种专有名词直接变 UNK。数据驱动的 CNN n-gram 反而比固定词表灵活。

### 阶段 5：BERT——97.27%，彻底碾压

最后一版换成了 `bert-base-chinese` + 分类头。102M 参数，只训了 2 epoch（早停），验证 F1=0.9753，测试 **97.27%**。

几个让我印象特别深的例子：

| 样本 | BiLSTM 最好版本 | BERT |
|------|:--------------:|:----:|
| "女排横扫巴西夺世界杯" | 游戏 ✗ | **体育 ✓ 95.9%** |
| "暗黑破坏神4新赛季" | 游戏 ✓ | **游戏 ✓ 99.9%** |
| "戴秉国会见奥巴马" | — | **时政 ✓ 99.8%** |
| "大盘调整/戴尔财报/机器人" | — | **全部正确 ✓** |

BERT 不是从 93% 爬到 95% 那种渐进式提升，是直接从"很多错的"变成"几乎全对"。**泛化能力碾压级别——预训练给的不是参数量，是"这个世界长什么样"的常识。**

#### 回过头看

五个迭代，每个都让我多理解一层。纯 BiLSTM（哦，它能分）→ +Attention（它能挑重点）→ +CNN（它能学搭配）→ +Jieba（固定词表有坑）→ BERT（预训练才是答案）。同一个数据集，五个不同的模型，我看到的是五个完全不同的"新闻分类"图景。

准确率不骗人，但会把人带偏。CNN+Attn 测试集 95.33% 比 BiLSTM 93.24% 高了 2 个点，交互样本反而更差——它把训练集的分布偏见学得更"透彻"了。如果不是自己一条一条试，我肯定以为 CNN 版本更好。

从头训练真的很难。11.8M 参数做到 93%，调参、调架构、处理 GPU 兼容、处理 Windows 死锁。BERT 102M 参数做到 97%，几乎不用调，搞清楚 `from_pretrained` 就行。"预训练"在 Week03 只是概念，跑了五版之后才真正知道它意味着什么。

还有 1080 Ti。后来 Week06 全量微调 Qwen 时果断上云，就是因为被 1080 Ti 搞怕了：峰值 <8GB 才安全、DataLoader 多进程就死锁、FP16 纯属浪费。这些坑不是课本上的，是 GPU 无响应以后 Ctrl+C 按出来的。

## 二、Week04：BERT——知识是可以迁移的

学了 BERT 的 Transformer Encoder 架构，搞懂了预训练→微调范式。然后把 Week03 的 cnews 分类从 BiLSTM 换成了 `bert-base-chinese` + 分类头。

### 从 95% 到 97%——不是边际提升，是质的区别

Week03 我改了五轮，BiLSTM 改到 CNN+Attn，测试集从 93% 慢慢爬到 95%。每一轮都要改架构、调参数、折腾 GPU——换两个百分点。

BERT 第一次跑：102M 参数，2 epoch，97.27%。

不光高了 2 个点——交互样本差得更远。CNN+Attn 把"女排横扫巴西"判成"游戏"，BERT 不但判对了，还给了 95.9% 的置信度。CNN+Attn 学的是训练集的分布偏置（长句 = 家居），BERT 知道"女排是一种体育比赛"。

### 预训练不是"提前训练"，是"已经知道这个世界什么样"

Week03 的 LSTM 从零开始。词向量随机初始化，模型不知道"国足"和"足球"有关系，不知道"女排"是一种运动。它得在 5 万条数据里慢慢摸索。但 5 万条对"理解语言"来说太少了。

BERT 不一样。bert-base-chinese 在几十亿字的语料上预训练过——它已经知道"女排"和"体育"的关联、"国足"和"世界杯"的关联。微调不是在重新学语言，是在唤醒已有的知识，告诉它怎么用在新闻分类上。

### Self-Attention 不神秘

第一次看到 Attention 公式的时候觉得特别复杂。Q、K、V 三个矩阵，线性变换、softmax。写代码的时候才发现，核心就三行：

```python
scores = Q @ K.T / sqrt(d_k)         # 每个词和其他词的相似度
weights = softmax(scores)             # 归一化成权重
output = weights @ V                  # 加权求和
```

矩阵乘法。就这么简单。但正是这个简单的矩阵乘法，每个词能直接看到所有其他词——不需要像 LSTM 那样靠隐状态一步一步传。LSTM 是"传纸条"，Transformer 是"开全员大会"。

### Word Embedding 到 WordPiece：词不是最小的单元

Week03 我按词分——"比特币"是整个词，"比特"是另一个词。BERT 用 WordPiece——"比特币"被切成 "比特"+"##币"，"比特率"被切成 "比特"+"##率"。

模型能学到"比特"是一个语义单元，可以跟"币""率"等后缀自由组合。我之前 3 万词表覆盖不了的专有名词（比如"比亚迪"），"比"+"##亚"+"##迪"三个子词搞定。

新闻分类里这个特别有用——"发改委大佬落马"在 BERT 里不再是一个整句，而是拆成"发改"+"##委"+"大佬"+"落马"，每个子词的语义 BERT 都已经学过。

#### 直接感受

从 LSTM 换到 BERT，最不习惯的是几乎不用调参。Week03 我花的时间：70% 调参 + 30% 理解架构。BERT：10% 调参 + 90% 理解架构。虽然 BERT 大了 10 倍（102M vs 11.8M），预训练已经把最难的部分做完了。

代价是单条推理从 ~10ms 变成 12-30ms。但 97.27% 的准确率下，多等几十毫秒没人在意。

## 三、Week05：插曲——从零写了一个 GPT

这一周偏离了分类主线，去写了一个自回归语言模型。但回头想，它对理解 Week06 的 LLM 方案帮助很大——如果没亲手写过 Causal Masking、没亲眼看到 temperature 怎么影响生成、没踩过 Teacher Forcing 的坑，后面做零样本和 SFT 的时候不可能那么快定位问题。

简单说就是：Week04 理解了 BERT 怎么"读"，Week05 搞清楚了 GPT 怎么"写"。BERT 是读完全句再判断，GPT 是走一步看一步。核心技术上没新东西——Multi-Head Attention 还是那个矩阵乘法，只是多了一行 `mask` 挡住未来。写完一遍之后，Transformer 算是彻底"刻进骨头里"了，后面 Week06 用 Qwen 做分类完全没心理障碍。

## 四、Week06：三种方案，两天十几组实验

TNEWS 数据集，15 类新闻标题，53K 训练 / 10K 验证。科技 5955 vs 证券 257，23:1 不均衡。对比了三条路线：BERT 微调、LLM 零样本、LLM SFT+LoRA，外加全量微调 Qwen 做基线。

两天里跑了 3 种池化策略、5 种 Loss、4 个 LoRA r 值、1 次解析器优化、1 次上云全量微调——加起来十几组对比。

### 先看数据再跑模型

在动任何代码之前，花了 5 分钟看数据分布。`explore_data.py` 告诉我三件事：

- 类别严重不均衡：科技 5955 vs 证券 257（23:1）
- 文本极短：平均 22 字，P99=39 字
- 15 个类之间有微妙差异："科技"vs"教育"、"财经"vs"证券"的标题用词很接近

从这三点出发，我心里有了三个待验证的判断：

1. `max_length=64` 足够（P99=39），不用 BERT 默认的 128/512
2. 不处理不均衡的话，模型会直接放弃小类
3. 池化策略要跑实验——短文本上 cls 不一定最好

看数据这个习惯是 Week03 养成的。当时 CNN+Attn 测试集分数更高但交互样本更差，就是因为没仔细看训练集的文体分布。这次不能再犯。

### BERT 池化：cls 被 mean 打脸

固定 epochs=3、batch_size=16、lr=2e-5，只改 `--pool`。三个命令，三个 checkpoint，分别评估。

| 池化 | val_acc | Macro F1 |
|---------|:-------:|:--------:|
| cls | 56.81% | 0.5538 |
| **mean** | **57.32%** | **0.5644** |
| max | 56.41% | 0.5597 |

**mean 比 cls 高。** 跑完第一反应是检查代码有没有写错——确认三遍没有 bug。

想了一下原因：TNEWS 标题平均 22 个字，短文本上每个词都承载了重要的分类信息。mean 对所有 token 做平均，保留了每个词的信号；[CLS] 虽然理论上做"句子摘要"，但短文本上优势不明显。

max 最差（56.41%）——只取最强特征，对"科技"vs"教育"这种需要细微区分的场景，丢失太多信息。这个倒符合预期。

教训很简单：跑之前我直觉觉得 cls 好。如果不跑这三个实验，我会一直这么认为。

```
epoch 1: train_acc=51.4%, val_acc=54.9%
epoch 2: train_acc=62.2%, val_acc=56.8% ← 最佳
epoch 3: train_acc=71.8%, val_acc=56.7% ← val_acc 停了，train_acc 还在涨
```

epoch 2 到 3，train_acc 跳了 10 个百分点但 val_acc 没动——多出来的那 10% 不是在学规律，是在背训练集答案。3 epoch 对这个任务已经到极限了。

### 五种 Loss 全对比

固定 BERT cls、epochs=3、batch_size=16，只改 loss_type。跑了 5 个版本。

| Loss | val_acc | Macro F1 | 证券 Recall | 科技 Recall |
|------|:-------:|:--------:|:----------:|:----------:|
| **Focal (gamma=2)** | **0.5724** | **0.5613** | 0.378 | 0.514 |
| 普通 | 0.5681 | 0.5480 | 0.178 | 0.524 |
| Soft 加权 (sqrt) | 0.5632 | 0.5574 | 0.600 | 0.500 |
| 硬加权 (balanced) | 0.5617 | 0.5575 | **0.622** | 0.449 |
| 两阶段 (freeze) | 0.5276 | 0.4932 | 0.000 | — |

这个表信息量很大：

Focal Loss 拿了 val_acc（0.5724）和 Macro F1（0.5613）双料冠军。它的"自动聚焦难分样本"机制对 15 类边界模糊的分类确实有效——不牺牲大类（科技 0.514），同时提升小类（证券 0.378），让整体更均衡。

加权方案在"救小类"上最强——证券 Recall 从 0.178 飙升到 0.622（硬加权）和 0.600（Soft）。代价是科技 Recall 从 0.524 分别降到 0.449 和 0.500。

两阶段冻结全面失败。冻结 BERT 后只练 11.5K 参数的分类头，val_acc 只有 0.5276，证券 Recall 直接为零。分类任务还是要微调让特征适配当前分类体系——光在顶上挂个分类头不够。

### 加权 Loss 消融：国际类比科技类还惨

`compare_class_weight.py` 的输出做了一个类别级别的 Recall 对比：

| 类别 | 样本数 | 普通 Recall | 加权 Recall | 变化 |
|------|:-----:|:----------:|:----------:|:---:|
| **证券**（最小） | 257 | 0.178 | 0.622 | **+0.444** |
| 故事 | 215 | 0.479 | 0.637 | +0.158 |
| 军事 | 716 | 0.472 | 0.620 | +0.148 |
| 财经 | 956 | 0.459 | 0.474 | +0.015 |
| 体育 | 767 | 0.718 | 0.709 | -0.009 |
| 科技（最大） | 5955 | 0.524 | 0.449 | -0.075 |
| **国际** | 905 | 0.592 | 0.428 | **-0.165** |

两个意外：

证券 Recall 从 0.178 跳到 0.622（+0.444），验证了加权对小类的强效。但**国际类 Recall 下降（-0.165）比科技类（-0.075）还多**。国际类有 905 条样本，不是小类。这说明加权方案的"牺牲"不均匀——与少样本类别语义接近的类更容易被误伤。国际和财经/科技的边界本身就模糊（"中美贸易"算国际还是财经？）。

整体效果：Macro F1 从 0.548 升到 0.558，Accuracy 从 0.568 降到 0.562。0.6% 的准确率换 1% 的 Macro F1。

### Soft 加权：试试别那么极端

硬加权下证券权重是 13.81 倍——觉得太极端了，国际类被误伤得厉害。试试平滑。

在 `compute_class_weight("balanced")` 基础上加了 `np.sqrt()`——证券从 13.81 降到 3.73，科技从 0.60 升到 0.77。改了一行代码。

| 指标 | 普通 | 硬加权 | Soft 加权 |
|------|:---:|:------:|:---------:|
| val_acc | 0.5681 | 0.5617 | **0.5632** |
| Macro F1 | 0.5480 | **0.5575** | 0.5574 |
| 证券 Recall | 0.178 | **0.622** | 0.600 |
| 科技 Recall | 0.524 | 0.449 | **0.500** |
| 国际 Recall | **0.592** | 0.428 | 0.464 |

Soft 介于两者之间：证券 Recall 0.600（十分接近 0.622），科技 Recall 0.500（远好于 0.449）。但国际 Recall 0.464——还是比普通的 0.592 差不少。硬加权的"过度牺牲"没有完全解决。

### 零样本：从 36% 到 48%，修的不是模型是解析器

第一版零样本：

```
样本数 : 200
准确率 : 36.0%
无法解析 : 58 条 (29.0%)
```

不到一半的准确率，接近三分之一无法解析。一开始以为是 Qwen 不行，看了具体输出才发现——它输出的内容大多数是合理分类，只是不在我的 15 个类别白名单里。

| Qwen 输出 | 应该归为 | 为什么没匹配 |
|-----------|---------|------------|
| "房地产" | 房产 | "房地产"和"房产"不同词 |
| "武器" | 军事 | 知识点粒度不同 |
| "政治" | 国际 | 分类体系不同 |
| "电影" | 娱乐 | 同上 |

根因：`parse_prediction` 只做了精确子串匹配——"房地产"不是 15 个类别名之一，直接返回 None。

改法：在 `parse_prediction` 加一个同义词映射表，把 Qwen 可能输出的常见词映射到标准类别名。

第一轮加约 30 个映射。再跑：
```
准确率 : 44.0%
无法解析 : 31 条 (15.5%)
```

又补了"交通"→汽车、"时尚"→娱乐、"美食"→文化等 30+ 个映射：

| 阶段 | 准确率 | 无法解析 |
|:----:|:-----:|:--------:|
| 原始 | 36.0% | 58 条 (29.0%) |
| 第一轮 | 44.0% | 31 条 (15.5%) |
| **第二轮** | **48.0%** | **3 条 (1.5%)** |

剩余 3 条："健康"(x2)、"自然灾害"(x1)——这三个确实不属于 15 个类别。

零样本的问题不是 Qwen 能力不够，是接口没对齐。LLM 的知识体系和 15 类的任务体系不是一一对应的。加 60 个同义词映射就能提 12 个百分点，说明 Qwen 其实知道这些类，只是输出形式跟解析器对不上。

### Few-shot 实验：技术路线错了比不做更惨

零样本 48% 之后，自然想到试试 Few-shot——给 Qwen 看几个示例，看它能不能学得更准。结果过程本身就有故事。

**第一轮：把示例塞进 System Prompt。** 用 `→` 箭头连起来："新闻：xxx → 体育"。

结果惨不忍睹——准确率跌到 32%，越改越差，差点跌到 16.5%。看了输出才发现搞笑的事情：Qwen 根本没在做分类，它在**模仿示例的格式**。看到示例里写了"新闻标题"，它就跟着输出"新闻标题：xxx"，完全不顾"只输出类别名"的指令。

**第二轮：改用多轮对话格式。** 这才是正确的 Few-shot——把示例做成 user/assistant 交替对话：

```
user: "新闻标题：国足晋级\n类别："
assistant: "体育"

user: "新闻标题：比特币突破\n类别："
assistant: "财经"

user: "新闻标题：[真实要分类的文本]\n类别："
```

Qwen 看到这种格式后，理解了"用户问→助手答类别名"的模式。同样的 3 条示例，错误做法 16.5%，正确做法 47.0%——**差了 30 个点。**

| k | 示例数 | 准确率 | 无法解析 |
|:-:|:------:|:-----:|:--------:|
| 0 | 0 | 48.0% | 1.5% |
| 1 | 15（每类1条） | 47.5% | 7.5% |
| **2** | **30（每类2条）** | **50.0%** | 7.5% |
| 3 | 45（每类3条） | 48.5% | 7.5% |

几个发现：

1. **Few-shot 的格式比内容重要得多。** 同样的 3 条示例，塞 System Prompt 崩到 16.5%，多轮对话拉到 47.0%。Qwen 是在对话数据上微调过的——它天生理解 user→assistant 的模式。把示例套进这个格式，它就能"悟"出来。

2. **和 LoRA r 消融一样的规律：k=2 最优，更多反而降。** 45 条示例给太多，模型反而混乱了——不同类的示例可能引入矛盾信号。

3. **Qwen2-0.5B 的天花板在 50%。** 零样本 48%，Few-shot 50%，给了 30 条示例只提 2 个点。追不上 BERT 和 LoRA 的 57%——这是模型规模本身的限制，不是 prompt 设计能突破的。

### SFT+LoRA r 值消融

固定 5K 训练数据、3 epochs、LoRA target_modules=q/k/v/o，只改 r 和 alpha。

| r | 可训参数 | 占比 | 准确率 |
|:-:|:--------:|:----:|:-----:|
| 4 | 0.55M | 0.11% | 53.5% |
| **8** | **1.1M** | **0.22%** | **57.0%** |
| 16 | 2.2M | 0.44% | 54.5% |
| 32 | 4.4M | 0.89% | 55.0% |

**r=8 是最优点。** r=4 不够力（53.5%），r≥16 不升反降（54.5%、55.0%）。

为什么 r 更大反而更差？5K 条数据对分类来说是"只改一点点"的任务——LLM 已经知道怎么用语言了，只需要在注意力层做微调。r=8 的 8 维子空间刚好够表达这种"微调"；r=16 多了多余的自由度，模型用这些额外参数去拟合训练集噪声而不是通用规律。

跟 Week03 的 CNN+Attn 问题是同一个性质——模型容量大于数据量时，模型会选择"记住"而不是"理解"。

### 全量微调 Qwen：500 倍参数，反而更差

动机很直接：LoRA 只动了 0.22% 的参数。全量 100% 一起训，会不会更好？

本地 1080 Ti 没法跑全量微调（内存不够），上 AutoDL 4090D。踩了个坑：`train_sft.py` 对模型路径做了 `Path(...).resolve()`，把 `"Qwen/Qwen2-0.5B-Instruct"` 转成了本地路径导致加载失败——改成直接用字符串解决。上云后还遇到了 `trust_remote_code=True` 触发 `generate.py` 从官方源拉取失败，本地上传模型打包解决。

| 方案 | 参数量 | 占比 | 准确率 |
|------|:-----:|:----:|:-----:|
| LoRA r=8 | 1.1M | 0.22% | **57.0%** |
| **全量微调** | **494M** | **100%** | **55.0%** |

500 倍参数，低 2 个百分点。

原因：494M 参数对 5K 条数据，参数是数据的 3000 倍以上。模型不需要理解分类规律，只需要记住这 5000 条。全量微调的优化空间是 494M 维，有无数种"背答案"的方式。LoRA 把优化空间限制在 1.1M 维的低秩子空间，模型被迫学通用规律。这后来我管它叫 **LoRA 的天然正则化**。

### 两天踩的坑

1. **`train_sft.py` 路径 bug**：`str(Path(args.model_path).resolve())` 把 HF 模型名（`Qwen/Qwen2-0.5B-Instruct`）转成了本地绝对路径导致 OSError。改成直接传字符串解决。
2. **本地 GPU 高温**：1080 Ti 跑 BERT 训练时温度飙到 90°C，风扇 100%。长任务（30min+）果断上云。
3. **云端网络**：AutoDL 上 HuggingFace 下载触发 `[Errno 99] Cannot assign requested address`。设 `HF_ENDPOINT=https://hf-mirror.com` 解决，但 `trust_remote_code` 的 `generate.py` 不受镜像影响，要本地上传完整模型。
4. **命名冲突**：把 `--use_class_weight` 改成 `--loss_type` 后，checkpoint 保存时还引用旧的 `args.use_class_weight`，AttributeError。修了一行。
5. **PowerShell 中文路径**：`cd` 到含中文路径时 PowerShell 编码出错，始终"找不到路径"。删掉脚本里的 `cd` 行，手动切目录解决。
6. **混淆矩阵 PNG 覆盖**：多次跑 `evaluate.py` 时 `confusion_cls.png` 被后跑的覆盖。改成每次评估后 `Copy-Item` 重命名保存。

### 汇总

| 实验 | 最佳配置 | 准确率 | 对比 |
|------|---------|:------:|------|
| 池化 | mean | **57.32%** | cls 56.81% |
| Loss | Focal | **57.24%** | plain 56.81% |
| 零样本（原始） | — | 36.00% | — |
| 零样本（优化） | +60 同义词 | **48.00%** | 原始 36% |
| Few-shot | k=2, 30条 | **50.00%** | 零样本 48% |
| LoRA | r=8, 5K | **57.00%** | BERT 56.81% |
| 全量微调 | — | 55.00% | LoRA 57.00% |

几个直接的结论：

1. BERT mean 池化 57.32%——最稳妥，参数刚好匹配数据量，推理只要 1ms。
2. LoRA r=8 用 0.22% 参数做到 57.0%，全量微调用 500 倍参数反而 55.0%——参数不是越多越好。
3. 零样本 36%→48% 的核心不是换模型，是修解析器。Few-shot 多轮对话做到 50%，但对 0.5B 小模型，边际收益已经到顶。
4. Focal Loss 是 val_acc 和 Macro F1 双料冠军，但如果目标是救小类 Recall 就选加权。没有"最好"的 Loss，只有最适合目标的 Loss。

## 五、跑了四周以后

### 同一个任务，五套做法

| 周次 | 方案 | 核心思路 |
|:----:|----------|---------|
| 3 | LSTM | 从头训，顺序建模 |
| 4 | BERT | 预训练后微调，双向上下文 |
| 5 | MyGPT | 自回归预训练，生成即分类 |
| 6 | 零样本 | 不训练，直接问 |
| 6 | SFT+LoRA | 只动 0.22% 参数，其余冻结 |

同样一批新闻标题，我可以用 LSTM 慢慢 fit、可以用 BERT 快速微调、可以拿 LLM 直接"问"。工具越往后越强，但越往后需要越多的工程判断才能用好。

### 参数多 ≠ 好

| 模型 | 参数 | 效果 |
|------|:---:|:----:|
| LSTM | ~1M | 不够 |
| BERT | 102M | 刚好（预训练过的） |
| LoRA r=8 | 1.1M | **57.0%**，性价比最高 |
| LoRA r=4 | 0.55M | 53.5%，容量不够 |
| LoRA r=16 | 2.2M | 54.5%，开始过拟合 |
| LoRA r=32 | 4.4M | 55.0%，继续降 |
| Qwen 全量 | 494M | 55.0%，5K 条数据白搭 |

四周来最大的认知变化：参数和数据要匹配。5K 条数据下，LoRA 的 1.1M 参数比 Qwen 全量的 494M 参数更有效——1.1M 没能力背答案，只能学通用规律。

### 实验结果反复打我脸

四周下来，至少三次"我以为"被打翻：

- 我以为 cls 最好 → mean 更高
- 我以为零样本 ~43% → 原始 36%（问题在解析器，不在模型）
- 我以为全量微调应该比 LoRA 好 → LoRA 高 2 个点
- 我以为 Few-shot 塞 System Prompt 就行 → 16.5%，差 30 个点

还有一个意外：Focal Loss，我本来没抱期望，结果是 val_acc 和 Macro F1 双料最高。

每次打脸学到的东西都比"验证猜测"多得多。这就是跑实验的意义。

### 四周前 vs 四周后

四周前，我只用"准确率"衡量模型好坏。现在我会看四件事：

- **准确率**：整体猜对比例，但会被大类主导
- **Macro F1 / Recall 分布**：模型有没有抛弃小类？
- **推理速度**：BERT 1ms vs LLM 50ms——线上部署 50 倍差距不能忽略
- **训练成本**：零样本 = 0，LoRA = 30min，全量 = 1h + 上云——成本不是线性的

### Loss 选型的决策经验

Focal Loss 适合"整体公平"——val_acc 和 Macro F1 双料最高，每个类都照顾到。加权方案适合"救急"——证券 Recall 能拉到 0.62。没有谁更好，看目标。两阶段冻结失败了，光靠 11.5K 的分类头顶不住。

---

**一句话收尾：**

做文本分类，模型不是越新越好，参数不是越多越好。理解你的数据、匹配你的资源、跑实验验证你的判断——这才是真的会做分类。

四周前我只知道"把标题塞进 LSTM 就能分类"。四周后我知道同一条标题，BERT 用 1ms 给 15 个类打分、零样本让 LLM 自由发挥、LoRA 用 0.22% 参数做到近似精度。没有哪个最好，只有哪个最合适。
